# Definition of Done לעבודת תוכן ופרוטוקול

## מטרה

המסמך הזה מגדיר טקס סגירה אחיד לכל שינוי תוכני, קליני, תפעולי או מבני בפרויקט.

המטרה היא שכל checkpoint עתידי ייסגר באותה מתכונת, בלי תלות בזיכרון מקומי או בפרשנות משתנה.

## עקרון מנחה

פער או תת-משימה אינם נסגרים רק כי "בוצע שינוי".

הם נסגרים רק כאשר יש:

1. שינוי בפועל
2. אימות מול מקור אמת
3. אימות מול המוצר
4. תיעוד מסודר
5. אם צריך, גם אימות build, test, deploy ו-production

## סוגי שינויים

### 1. שינוי תוכן/פרוטוקול

לדוגמה:

- ניסוח צומת
- סדר צעדים
- הוספת אנמנזה
- התאמה ל-BLS
- הוספת מקור
- הוספת ענף קליני

### 2. שינוי UX / ניווט / מבנה מוצר

לדוגמה:

- שינוי מסגור UI
- תפריט ניווט
- קפיצה לסכמה
- פיצול צומת
- שינוי היררכיה

### 3. שינוי תיעודי בלבד

לדוגמה:

- מיפוי מקור אמת
- מסמך מעקב
- checklist
- DoD

## Checklist סגירה קבוע

יש לעבור על כל הסעיפים הרלוונטיים ולסמן פנימית שהושלמו:

1. הוגדר במדויק מה הפער או המשימה שנסגרים.
2. בוצע שינוי בפועל בקוד, ב-JSON, או במסמך הרלוונטי.
3. בוצעה השוואה למקור האמת הרלוונטי.
4. בוצע אימות ידני של הצומת, המסלול, המסך או המסמך שהשתנה.
5. אם השינוי נוגע לקוד ריצה:
   - `npm run build`
   - `npm test -- --runInBand`
6. אם השינוי נוגע למה שנפרס לפרודקשן:
   - `git push origin main`
   - אימות ש-Cloudflare / alias עודכנו
   - אימות שהפרודקשן מגיש את ה-bundle הצפוי או נשאר תקין כמצופה
7. עודכן `docs/tracking/protocol-gap-tracker.md`.
8. אם השינוי משמעותי לפרויקט:
   - עודכן `current.md`
   - עודכן `tasks.md`
9. נכתב סיכום קצר:
   - מה שונה
   - איך נבדק
   - מה נשאר פתוח

## Sanity Checks מחייבים אחרי כל פיצ'ר

הסעיף הזה נוסף כדי למנוע מצב שבו פיצ'ר נסגר רק כי הקוד "נראה מוכן", למרות שבפועל הוא לא עובד או שבר משהו אחר.

כל פיצ'ר, תיקון או שינוי UX שמסומן כ-`done` חייב לעבור את sanity checks הבאים:

### 1. בדיקת חיבור אמיתי

אם הפיצ'ר תלוי ב-API, auth, DB, storage, env, route או service חיצוני:

1. לוודא שהכתובת שה-client משתמש בה היא הכתובת הנכונה בפועל.
2. לוודא שה-service באמת רץ ומאזין.
3. לוודא שהקריאה מחזירה payload מהסוג הצפוי, ולא fallback, HTML, 404, 500 או route אחר.
4. אם מדובר בפרודקשן, לבדוק את ה-endpoint החי עצמו ולא רק את ה-UI שמעליו.

### 2. בדיקת מסלול חי מקצה לקצה

לא מספיק לבדוק שהקומפוננטה נראית טוב.

צריך לעבור לפחות פעם אחת את ה-flow האמיתי שהמשתמש אמור לבצע:

1. כניסה למסך
2. אינטראקציה עיקרית
3. שמירה / שליחה / מעבר שלב / טעינה
4. אימות שהתוצאה חזרה למסך

אם הפיצ'ר תומך ב-create/update/delete או submit:

- חובה לבדוק לפחות פעולה אמיתית אחת end-to-end.

### 3. בדיקת טעינה ורענון

כל פיצ'ר עם state, טעינה, API או persistence חייב לעבור:

1. טעינה ראשונית
2. רענון מסך / reload
3. מצב ריק אם רלוונטי
4. מצב שגיאה אם אפשר לבדוק בצורה סבירה

### 4. בדיקת אי-שבירה רוחבית

אחרי כל פיצ'ר צריך לבדוק שלא נשברו אזורים סמוכים או תשתיות משותפות.

מינימום חובה:

1. המסך או הצומת ששונה
2. המסך הקודם והבא ב-flow
3. כל רכיב shared שנוגע לשינוי
4. אם השינוי נוגע ל-env / API / routing - לבדוק לפחות endpoint או flow נוסף שלא היה יעד השינוי

### 5. בדיקת Local מול Production

אם זה פיצ'ר שמיועד להיות פעיל בפרודקשן:

1. לאמת שהוא עובד ב-local או בסביבת dev אמיתית
2. לדחוף לענן
3. לאמת שהפרודקשן באמת קיבל את השינוי
4. לאמת שגם בפרודקשן הפיצ'ר באמת עובד, לא רק שה-bundle התחלף

### 6. בדיקת Config / Env

כל שינוי שעלול להיות מושפע מ-config חייב לכלול בדיקה מפורשת של:

1. `.env` בפועל
2. `.env.example` אם צריך
3. fallback בקוד
4. כתובת/פורט/host אמיתי
5. האם ה-build מגלם ערך שגוי בתוך bundle הפרודקשן

### 7. כלל חסימה לסגירה

אסור להעביר פיצ'ר או פער ל-`done` אם אחד מהבאים נכון:

1. בדקנו רק UI ולא את החיבור האמיתי
2. בדקנו רק local ולא production למרות שהפיצ'ר פרודקשני
3. בדקנו רק build/test בלי flow משתמש אמיתי
4. יש fallback לא ברור, env לא ודאי, או endpoint שלא אומת
5. יש סימן שהפיצ'ר "קיים טכנית" אבל לא עובד end-to-end

במקרים כאלה הסטטוס צריך להיות:

- `in_progress`
- או `blocked`

אבל לא `done`

## מינימום חובה לפי סוג שינוי

| סוג שינוי | מקור אמת | בדיקת UI/מסמך | Build/Test | Deploy/Prod | עדכון מעקב |
|---|---|---|---|---|---|
| תוכן/פרוטוקול | חובה | חובה | חובה | חובה | חובה |
| UX/ניווט/מבנה | לפי הצורך, ואם קשור לתוכן אז חובה | חובה | חובה | חובה | חובה |
| תיעוד בלבד | אם המסמך מבוסס מקור, חובה | חובה למסמך עצמו | לא חובה | לא חובה | חובה |

## ראיות סגירה שחייבות להופיע בסיכום

בכל סגירת משימה צריך להיות אפשר להצביע על:

1. קובץ או קבצים ששונו
2. בדיקה שבוצעה
3. סטטוס deploy אם היה deploy
4. מה נשאר פתוח אחריה
5. sanity checks שבוצעו בפועל
6. האם אומת flow משתמש אמיתי
7. האם אומת שהשינוי לא שבר flow סמוך או endpoint קשור

## נוסח סגירה מומלץ

בכל checkpoint עתידי נשמור על מבנה קבוע:

1. מה עשיתי
2. מה בדקתי
3. מה נדחף לענן
4. איך אימתתי פרודקשן
5. איזה sanity checks בוצעו
6. מה נשאר

## כלל סגירה למסמך המעקב

שורת פער ב-`protocol-gap-tracker.md` יכולה לעבור ל-`done` רק אם:

1. סעיף `מה נחשב תוקן` עודכן כך שישקף את המצב החדש
2. סעיף `איך מאמתים` מתאר בפועל את דרך האימות ששימשה
3. לוג העדכונים קיבל שורת סגירה
4. הפער ירד מרשימת השלבים הפתוחים

## מצב נוכחי

המסמך הזה הוא כעת מקור ה-DoD הרשמי של הפרויקט לכל עבודת תוכן, UX, פרוטוקול ותיעוד.
