# Definition of Done לעבודת תוכן ופרוטוקול

## מטרה

המסמך הזה מגדיר טקס סגירה אחיד לכל שינוי תוכני, קליני, תפעולי או מבני בפרויקט.

המטרה היא שכל checkpoint עתידי ייסגר באותה מתכונת, בלי תלות בזיכרון מקומי או בפרשנות משתנה.

## עקרון מנחה

פער או תת-משימה אינם נסגרים רק כי "בוצע שינוי".

הם נסגרים רק כאשר יש:

1. שינוי בפועל
2. אימות מול מקור אמת
3. אימות מול המוצר
4. תיעוד מסודר
5. אם צריך, גם אימות build, test, deploy ו-production

## סוגי שינויים

### 1. שינוי תוכן/פרוטוקול

לדוגמה:

- ניסוח צומת
- סדר צעדים
- הוספת אנמנזה
- התאמה ל-BLS
- הוספת מקור
- הוספת ענף קליני

### 2. שינוי UX / ניווט / מבנה מוצר

לדוגמה:

- שינוי מסגור UI
- תפריט ניווט
- קפיצה לסכמה
- פיצול צומת
- שינוי היררכיה

### 3. שינוי תיעודי בלבד

לדוגמה:

- מיפוי מקור אמת
- מסמך מעקב
- checklist
- DoD

## Checklist סגירה קבוע

יש לעבור על כל הסעיפים הרלוונטיים ולסמן פנימית שהושלמו:

1. הוגדר במדויק מה הפער או המשימה שנסגרים.
2. בוצע שינוי בפועל בקוד, ב-JSON, או במסמך הרלוונטי.
3. בוצעה השוואה למקור האמת הרלוונטי.
4. בוצע אימות ידני של הצומת, המסלול, המסך או המסמך שהשתנה.
5. אם השינוי נוגע לקוד ריצה:
   - `npm run build`
   - `npm test -- --runInBand`
6. אם השינוי נוגע למה שנפרס לפרודקשן:
   - `git push origin main`
   - אימות ש-Cloudflare / alias עודכנו
   - אימות שהפרודקשן מגיש את ה-bundle הצפוי או נשאר תקין כמצופה
7. עודכן `docs/tracking/protocol-gap-tracker.md`.
8. אם השינוי משמעותי לפרויקט:
   - עודכן `current.md`
   - עודכן `tasks.md`
9. נכתב סיכום קצר:
   - מה שונה
   - איך נבדק
   - מה נשאר פתוח

## מינימום חובה לפי סוג שינוי

| סוג שינוי | מקור אמת | בדיקת UI/מסמך | Build/Test | Deploy/Prod | עדכון מעקב |
|---|---|---|---|---|---|
| תוכן/פרוטוקול | חובה | חובה | חובה | חובה | חובה |
| UX/ניווט/מבנה | לפי הצורך, ואם קשור לתוכן אז חובה | חובה | חובה | חובה | חובה |
| תיעוד בלבד | אם המסמך מבוסס מקור, חובה | חובה למסמך עצמו | לא חובה | לא חובה | חובה |

## ראיות סגירה שחייבות להופיע בסיכום

בכל סגירת משימה צריך להיות אפשר להצביע על:

1. קובץ או קבצים ששונו
2. בדיקה שבוצעה
3. סטטוס deploy אם היה deploy
4. מה נשאר פתוח אחריה

## נוסח סגירה מומלץ

בכל checkpoint עתידי נשמור על מבנה קבוע:

1. מה עשיתי
2. מה בדקתי
3. מה נדחף לענן
4. איך אימתתי פרודקשן
5. מה נשאר

## כלל סגירה למסמך המעקב

שורת פער ב-`protocol-gap-tracker.md` יכולה לעבור ל-`done` רק אם:

1. סעיף `מה נחשב תוקן` עודכן כך שישקף את המצב החדש
2. סעיף `איך מאמתים` מתאר בפועל את דרך האימות ששימשה
3. לוג העדכונים קיבל שורת סגירה
4. הפער ירד מרשימת השלבים הפתוחים

## מצב נוכחי

המסמך הזה הוא כעת מקור ה-DoD הרשמי של הפרויקט לכל עבודת תוכן, UX, פרוטוקול ותיעוד.
