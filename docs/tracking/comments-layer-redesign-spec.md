# Comments Layer Redesign Spec

## סטטוס מסמך

- תאריך: `2026-07-08`
- סטטוס: `approved-partially-implemented`
- מבוסס על:
  - דרישות משתמש ישירות
  - מימוש קיים ב-`CommentsThread`, `CommentItem`, `CommentForm`
  - מיקום קיים ב-`StepByStepView`

## למה בכלל משנים

שכבת הקהילה הקיימת נבנתה כ"שכבת הבהרות מקצועית" עם טפסים, taxonomy, prompts ו-empty states עשירים.

בפועל, הכיוון הרצוי עכשיו שונה:

1. התגובות לא אמורות להוביל את המסך.
2. התגובות לא אמורות להיראות כמו כלי mentorship כבד.
3. התגובות אמורות להרגיש כמו שכבת שיחה פשוטה, מוכרת ומהירה בסגנון `Twitter/X`.
4. הזרימה הראשית של הפרוטוקול חייבת להישאר ראשונה, קשיחה ונקייה מהסחות.

## מטרת המוצר

ליצור שכבת תגובות משנית, פשוטה וברורה, שמאפשרת:

1. להוסיף הערה על צומת ספציפי בפרוטוקול.
2. לשרשר תגובה לתגובה קיימת.
3. לעשות `like`.
4. לצפות בדיון גם בלי להתחבר.
5. להגיב רק אחרי התחברות עם חשבון Google אמיתי.
6. לאפשר למנהל המערכת למחוק כל תגובה במקרה של spam / trolling / תוכן לא ראוי.
7. להציג מדדי שימוש בסיסיים כמו מספר צפיות לכל תגובה.
8. לשמור על MVP פשוט: ללא עריכת תגובות בגרסה הראשונה.

## עקרונות מוצר מחייבים

### 1. הזרימה הראשית קודמת לכל

שכבת התגובות תופיע רק אחרי:

1. תוכן הצומת
2. אזור `מה הצעד הבא`

אסור לה להפריע לתהליך קבלת ההחלטה בזמן שימוש בפרוטוקול.

### 2. תגובות הן שכבת תוספת, לא שכבת לימוד ראשית

התגובות לא מחליפות:

- דגשי מדריך
- טעויות נפוצות
- שאלות חזרה
- מקורות

הן רק מוסיפות שיח קהילתי סביב התוכן.

### 3. תחושת שימוש מוכרת

ה-UI צריך להרגיש כמו feed תגובות פשוט:

- תגובה ראשית
- שרשור תגובות
- לייק
- מונה תגובות
- מונה צפיות
- חיווי מי מחובר ויכול להגיב

לא כמו טופס קליני מורכב.

### 4. אחריות והרשאות ברורות

- כל אחד יכול לקרוא תגובות.
- רק משתמש מחובר עם Google יכול לעשות `like`.
- רק משתמש מחובר עם Google יכול לכתוב תגובה או תגובת-משנה.
- רק אדמין יכול למחוק כל תגובה.
- משתמש רגיל יכול למחוק את התגובה של עצמו.

## מצב קיים מול כיוון חדש

### מצב קיים

1. שכבת הקהילה בנויה כ-panel עשיר עם כותרות, badges, סטטוסים ו-seed prompts.
2. טופס הכתיבה דורש בחירת סוג תגובה (`שאלה`, `הבהרה`, `טיפ ביצועי` וכו').
3. יש fallback של `guest login` ברמת האפליקציה.
4. אין `likes`.
5. אין `view count`.
6. אין מודל UI פשוט בסגנון feed חברתי.

### מצב יעד

1. אזור תגובות קומפקטי ופשוט בתחתית כל צומת.
2. טופס כתיבה קצר וישיר ללא taxonomy חובה.
3. שרשור תגובות עד עומק מוגדר.
4. `like` לכל תגובה.
5. `view count` לכל תגובה.
6. CTA ברור להתחברות Google עבור כתיבה בלבד.
7. כלי ניהול מחיקה זמין תמיד לאדמין.

## חוויית משתמש רצויה

## 1. משתמש לא מחובר

המשתמש יכול:

- לראות תגובות
- לראות replies
- לראות like count
- לראות view count
- לא לעשות like

המשתמש לא יכול:

- לכתוב תגובה
- לשרשר תגובה
- לעשות like

במקום טופס פתוח יוצג CTA קצר:

`התחבר עם Google כדי להגיב`

## 2. משתמש מחובר

המשתמש יכול:

- לפרסם תגובה חדשה
- להשיב לתגובה
- לעשות like
- לראות את הפעילות של הדיון

הטופס צריך להיות קצר, נעים ומהיר:

- שדה טקסט
- כפתור פרסום
- חיווי שהתגובה תופיע מתחת לכל התוכן ולא תשפיע על רצף הפרוטוקול

## 3. אדמין

אדמין יכול:

- למחוק כל תגובה
- למחוק כל reply
- לראות את אותה שכבה כמו כולם בלי dashboard נפרד בשלב הראשון

אם נידרש בהמשך, נוסיף moderation dashboard נפרד. זה לא חלק מה-scope הראשוני.

## מבנה UI מוצע

## כותרת האזור

כותרת פשוטה ולא דרמטית:

- `תגובות על הצומת`
- subtitle קצר:
  - `דיון קהילתי משני לפרוטוקול`

## סדר האזור

1. כותרת + מונה תגובות כללי
2. CTA כתיבה / התחברות
3. רשימת תגובות ראשיות
4. לכל תגובה:
- avatar / initials
- שם
- זמן
 - תוכן
  - שורת actions: `like`, `reply`, `delete`
  - metadata קטן: `time`, `likes`, `views`
   - delete אם יש הרשאה
5. replies יופיעו מוזחות מתחת לתגובה

## מה לא יהיה בגרסה הזו

- taxonomy של סוגי תגובה
- seed prompts
- cards של "איך פותחים דיון טוב"
- empty state חינוכי ארוך
- badges רבים
- סטטוס "מחובר לדיון"
- עומס ויזואלי של "שכבת קהילה"

## החלטות מוצר שדורשות יישור במימוש

### 1. ביטול guest login לשכבת התגובות

הדרישה החדשה סותרת את ההנחה הקודמת של `guest login fallback`.

לכן, עבור שכבת התגובות:

1. אורח יכול לצפות בלבד.
2. תגובה ו-reply דורשים Google login בלבד.
3. גם `like` דורש Google login בלבד.

החלטה טכנית נלווית:

- אם `guest login` נשאר באפליקציה לצרכים אחרים, צריך לוודא שהוא לא פותח הרשאת comment בטעות.

### 2. like לאורחים

הוחלט שלייקים יהיו רק למחוברים.

המשמעות:

1. אין צורך בפתרון guest-like ב-MVP.
2. אפשר להישען על זהות המשתמש המחובר למניעת duplicate likes.
3. כפתור like לאורח יוצג כ-disabled או יפעיל CTA להתחברות.

### 3. view count לתגובה

יש להגדיר מה נחשב צפייה:

הגדרת עבודה מומלצת:

- צפייה נספרת כאשר תגובה נכנסה ל-viewport למשך זמן מינימלי.
- אותו מכשיר לא יעלה count שוב ושוב בכל refresh בטווח זמן קצר.

החלטת UX:

- `view count` יוצג לכולם, אבל כ-`metadata קטן` ולא כמספר בולט.

### 4. edit לתגובה

הוחלט שבגרסה הראשונה אין עריכת תגובות.

MVP כולל רק:

1. כתיבת תגובה
2. תגובת-שרשור
3. like למחוברים
4. מחיקה עצמית
5. מחיקה גלובלית לאדמין

## שינויים נדרשים בנתונים

## Comment model

המודל הקיים כולל:

- `id`
- `nodeId`
- `content`
- `authorId`
- `parentCommentId`
- `createdAt`
- `updatedAt`

צריך להרחיב אותו או להוסיף טבלאות קשורות עבור:

1. `likesCount` או relation ל-`CommentLike`
2. `viewsCount` או relation ל-`CommentView`
3. שדות moderation אם נרצה soft delete בעתיד

### מבנה מומלץ

1. `Comment`
2. `CommentLike`
3. `CommentView`

כך נשמור על auditability ונוכל למנוע ספירה שגויה.

## API נדרש

### קיים

- `GET /api/comments/:nodeId`
- `POST /api/comments`
- `PUT /api/comments/:commentId`
- `DELETE /api/comments/:commentId`

### נדרש להוסיף או לשנות

1. `POST /api/comments/:commentId/like`
2. `DELETE /api/comments/:commentId/like` או `POST toggle-like`
3. `POST /api/comments/:commentId/view`
4. החזרת `likesCount`, `viewerHasLiked`, `viewsCount`, `replyCount`
5. אכיפת Google-auth-only ב-create/reply
6. אכיפת Google-auth-only ב-like
7. אכיפת admin delete all

## sanity checks מחייבים לשכבה הזו

1. תגובות מופיעות רק מתחת ל-`מה הצעד הבא`.
2. משתמש לא מחובר לא רואה composer פתוח אלא CTA להתחברות.
3. משתמש Google מחובר יכול לפרסם תגובה ראשית ו-reply.
4. אורח לא יכול לעשות like ומקבל CTA להתחברות.
5. like לא נספר פעמיים לאותו משתמש.
6. view count לא מטפס בטעות על refresh אינסופי.
7. אדמין יכול למחוק כל תגובה.
8. משתמש רגיל יכול למחוק את שלו ולא יכול למחוק של אחרים.
9. אין אפשרות edit ב-UI ואין endpoint פתוח לעריכה ב-MVP.
10. שרשור לא שובר מובייל.
11. הוספת שכבת התגובות לא שוברת את רצף `S -> התרשמות זירה -> AVPU`.

## Definition of Done

הפער ייחשב סגור כאשר:

1. שכבת התגובות מופיעה בתחתית האזור, אחרי `מה הצעד הבא`.
2. העיצוב מרגיש כמו feed תגובות פשוט ולא כמו בלוק mentorship מורכב.
3. תגובה ו-reply זמינים רק אחרי Google login.
4. like זמין רק אחרי Google login ועובד פעם אחת למשתמש.
5. view count מוצג כ-metadata קטן ועובד לפי הגדרה מדידה.
6. משתמש רגיל יכול למחוק את שלו בלבד.
7. אדמין יכול למחוק כל תגובה בפועל.
8. אין edit ב-MVP.
9. הבדיקות הידניות והאוטומטיות עוברות.
10. המסמכים בכספת ובריפו מעודכנים לאותו מצב.

## הערות פתוחות

1. אם נרצה בעתיד edit, צריך להוסיף אותו כשלב שני ולא כחלק מה-MVP.
2. אם נרצה בעתיד like לאורחים, צריך להגדיר anti-abuse policy חדשה.
